# 迁移指南

本指南提供了 Solidity（以太坊的智能合约语言）与 Leo（Aleo 用于零知识应用的编程语言）之间的全面对比。文档涵盖了这两种语言之间的根本差异，并提供了实际示例来帮助从以太坊开发过渡到 Aleo 开发的开发者。

## 执行模型

Leo 和 Solidity 在执行方式上有根本性的不同：

  * **Ethereum/Solidity** ：链上执行，所有计算都在区块链上进行
  * **Aleo/Leo** ：使用零知识证明的链下执行，计算在私下完成，仅证明在链上验证


这种差异影响了许多下面可以看到的语言设计决策。

## 基本结构

### 头文件

每种语言中程序的开头不同：

**Solidity Contract:**


    // SPDX-License-Identifier: MIT
    pragma solidity ^0.8.19;

    contract MyToken {
        // Contract implementation
    }

**Leo Program:**


    // No license identifier required
    // No pragma needed

    program my_token.aleo {
        // Program implementation
    }

**关键差异：**

  * Leo 不需要许可证标识符或 pragma 语句
  * Leo 中的 import 语句使用程序 ID 而非文件路径
  * Leo 程序名称必须带有 `.aleo` 后缀
  * 在 Solidity 中定义合约使用 `contract` 关键字。在 Leo 中，程序使用 `program` 关键字定义，后跟花括号 `{}`


### 构造函数

Solidity 中的构造函数是可选的。Leo 目前没有构造函数，但 [ARC-0006: Program Upgradability](<https://github.com/ProvableHQ/ARCs/discussions/94>) 将为 Leo 新程序引入必须的构造函数。Solidity 中的构造函数仅运行一次，而 Leo 中的构造函数是不可变的，将作为部署或升级的一部分运行，因此用于定义程序的可升级逻辑。

### 注释

两种语言支持相同的注释语法：


    // 两种语言中的单行注释

    /*
     * 多行注释
     * 工作方式也相同
     */

### 导入

**Solidity 的导入系统：**


    // 导入整个文件
    import "./MyContract.sol";

    // 导入特定符号
    import {Symbol1, Symbol2} from "./MyContract.sol";

    // 使用别名导入
    import * as MyAlias from "./MyContract.sol";

    // 从 node_modules 导入
    import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

**Leo 的导入系统：**


    import credits.aleo;

    program helloworld.aleo {
        // Program implementation
    }

**重要：Leo 的 import 还必须在`program.json` 中声明：**


    {
      "program": "my_token.aleo",
      "version": "0.0.0",
      "description": "",
      "license": "MIT",
      "dependencies": [
        {
          "name": "credits.aleo",
          "location": "network",    // Importing from network
          "network": "testnet"
        },
        {
          "name": "board.aleo",
          "location": "local",      // Importing from local
          "path": "../board"
        }
      ]
    }

**依赖类型：**

  * **网络依赖** ：已部署在 Aleo 网络（主网、测试网）上的程序
  * **本地依赖** ：本地文件系统目录中的程序


**关键导入差异：**

  * **Solidity** ：导入时直接将代码复制到你的合约中，在编译时合并到一个文件
  * **Leo** ：导入的程序保持独立实体，不会与当前文件合并，每个程序在区块链上保留自己唯一的程序 ID


## 数据与状态

### 状态变量与存储模型

Solidity 和 Leo 在处理状态变量、存储和数据隐私方面有显著差异。

**Solidity 的方式：**


    contract Storage {
        // 带可见性修饰符的状态变量（仅访问控制）
        uint256 public constant FIXED_VALUE = 100;  // 编译时常量
        uint256 public immutable RUNTIME_VALUE;     // 在构造函数中设置
        uint256 public permanentData;               // 自动生成 getter 函数
        uint256 internal shared;                    // 可在派生合约中访问
        uint256 private local = 42;                 // 不可在派生合约中访问，但链上仍可见

        constructor(uint256 _value) {
            RUNTIME_VALUE = _value;  // 可在构造时设置
        }

        function processData(uint256 tempData) public {
            uint256 memoryVar = tempData * 2;   // Memory (temporary)
            permanentData = memoryVar;          // Persisted to storage
        }
    }

**Leo 的方式：**


    program storage.aleo {
        // 仅编译时常量
        const FIXED_VALUE: u64 = 100u64;

        // 永久公开状态：所有人可访问
        mapping balances: address => u64;

        // 永久私有状态：存储在 record 中（具有密码学隐私）
        record Token {
            owner: address,
            amount: u64,
        }

        // transition 的参数和返回值默认为私有，除非标记为 public
        fn process_data(public amount: u64) -> (Token, Final) {
            let amount_loc: u64 = amount * 2u64; // Local variable

            let token: Token = Token {
                owner: self.caller,
                amount: amount_loc,
            };

            return (token, final {
                let current_balance: u64 = Mapping::get_or_use(balances, self.caller, 0u64);
                Mapping::set(balances, self.caller, current_balance + amount_loc);
            });
        }
    }

**关键差异：**

  * **Solidity 可见性** ：`private`、`internal`、`public` 控制代码访问权限，但所有数据在链上可见
  * **Leo 隐私** ：`public` 与 `private` 决定了实际的密码学隐私级别，即数据是存储在链上还是链下的 [records](../../../concepts/fundamentals/records/index.mdx) 中
  * **常量** ：Solidity 有 `constant` 和 `immutable` 两种，Leo 只有编译时的 `const`
  * **局部变量** ：`let` 关键字声明临时计算变量（类似于 Solidity 的 memory）
  * **不支持瞬态存储** ：Leo 不支持 Solidity 的 transient storage 概念


### Getter 函数

Solidity 编译器会自动为所有 `public` 状态变量创建 getter 函数。Leo 没有类似功能，而是提供直接的 API 访问来从节点查询 mapping 值，无需在程序代码中编写 getter 函数。


    // Solidity：自动生成 getter
    contract Example {
        uint256 public value;  // 自动创建 value() 函数作为 getter
    }

**关键差异：**

  * **Solidity** ：需要 public 状态变量或显式的 getter 函数才能从外部访问
  * **Leo** ：mapping 值可通过 REST API 端点直接访问，无需任何程序代码：
    * 端点：`GET /{network}/program/{programID}/mapping/{mappingName}/{mappingKey}`
    * 示例：`GET /testnet3/program/credis.aleo/mapping/account/aleo1abc...`


### 删除状态变量

**Solidity 的 Delete 操作：**


    contract DeleteExample {
        uint256 public value = 100;
        mapping(address => uint256) public balances;

        function resetValue() public {
            delete value;  // 重置为初始值（uint256 为 0）
        }

        function removeBalance(address user) public {
            delete balances[user];  // 删除 mapping 条目，重置为初始值（0）
        }
    }

**Leo 的 mapping 操作：**


    program delete_example.aleo {
        mapping balances: address => u64;

        fn remove_balance(user: address) -> Final {
            return final {
                // 删除前检查 mapping 是否包含该键
                let exists: bool = Mapping::contains(balances, user);
                if exists {
                    // 删除 mapping 条目
                    Mapping::remove(balances, user);
                }
            };
        }
    }

**关键差异：**

  * **Solidity** ：`delete` 操作符将变量重置为初始值（数字为 0，布尔值为 false，数组为空）
  * **Leo** ：仅支持使用 `Mapping::remove()` 删除 mapping 中的条目
  * **无直接 Delete** ：Leo 没有直接等价于 Solidity 的 `delete` 操作符用于其他变量类型
  * **专注于 Mapping** ：Leo 的删除功能专注于 mapping 条目，这是主要的状态存储机制
  * **键存在性检查** ：Leo 提供 `Mapping::contains()` 用于在操作前检查 mapping 中是否存在某个键


### 数据类型

类型系统在多个方面不同：

**常见类型（行为相似）：**


    // Solidity
    bool flag = true;
    uint32 smallNumber = 100;
    address userAddr = 0x742d35...;


    // Leo
    let flag: bool = true;
    let small_number: u32 = 100u32;  // Must append type suffix (u32)
    let user_addr: address = aleo1abc...;

**类型后缀要求：** 在 Leo 中，所有字面量必须显式附加其类型后缀。

**Leo 的特殊类型：**

Leo 引入了几个 Solidity 中不存在的独特密码学类型：


    // Leo 的独特密码学类型
    let field_element: field = 123field;          // Field elements for arithmetic
    let group_element: group = group::GEN;        // Group elements for elliptic curves
    let scalar_value: scalar = 456scalar;         // Scalar values for cryptographic operations
    let user_signature: signature = sign1...;     // Built-in signature type

**签名验证** ：Leo 的 `signature` 类型使用 Schnorr 签名方案（不同于 Solidity 的 ECDSA），签名使用 nonce、challenge hash 和私钥生成，然后通过根据公钥重建 challenge 来验证。

**Leo 的限制：**

  1. **整数限制：**




    // Solidity：支持最多 256 位
    uint256 bigNumber = 115792089237316195423570985008687907853269984665640564039457584007913129639935;


    // Leo：最多 128 位
    let big_number: u128 = 340282366920938463463374607431768211455u128;
    // uint256 equivalent does not exist

  2. **Leo 不支持的类型：**




    // Solidity 类型中 Leo 不支持的部分
    string memory text = "Hello World";           // 字符串类型在 Leo 中不可用
    bytes memory data = hex"1234";                // bytes 类型在 Leo 中不可用
    uint256[] memory dynamicArray;                // 动态数组在 Leo 中不可用
    mapping(string => uint256) stringMap;         // mapping 中字符串键在 Leo 中不可用
    enum Status { Active, Inactive }              // 枚举类型在 Leo 中不可用


    // Leo 的替代方案或变通方法
    let text_hash: field = 123456field;           // Convert text into Leo supported types such as Field
    let static_array: [u32; 5] = [1u32, 2u32, 3u32, 4u32, 5u32]; // Fixed-size arrays only
    mapping hash_map: field => u64;               // Use field for complex key types
    const ACTIVE: u8 = 0u8;                       // Use constants instead of enums
    const INACTIVE: u8 = 1u8;

  3. **Leo 中的静态数组：**




    // Solidity：支持动态和静态数组
    uint256[5] public staticArray;

    function updateArray() public {
        staticArray[0] = 42;  // 直接元素修改
    }


    // Leo：仅支持静态数组
    fn simple_array() -> [u32; 3] {
        // 创建包含 3 个元素的固定大小数组
        let numbers: [u32; 3] = [1u32, 2u32, 3u32];

        // 使用显式索引类型（u8）修改元素
        numbers[0u8] = 10u32;
        numbers[1u8] = 20u32;

        return numbers;
    }

**静态数组关键特性：**

  * **固定大小** ：数组大小必须在编译时确定
  * **元素访问** ：使用显式的索引类型（如 `array[0u8]`）
  * **无动态操作** ：不支持 `push()`、`pop()` 或调整大小等操作


### 类型转换

**Solidity 的隐式转换：**


    uint8 small = 100;
    uint256 big = small;  // Automatic conversion

    // Solidity 允许截断
    uint16 large = 300;
    uint8 truncated = uint8(large);  // Only the rightmost 8 bits are kept

**Leo 的显式转换：**


    let small: u8 = 100u8;
    let big: u32 = small as u32;  // Explicit casting required

    // Leo 在截断时会失败（检查式转换）
    let large: u32 = 300u32;
    let truncated: u8 = large as u8;  // Program fails, no truncation allowed unless able to fit into smaller size

    // 安全转换到更大的类型
    let safe: u64 = small as u64;    // No truncation

**关键差异：**

  * Leo 需要使用 `as` 关键字进行显式类型转换
  * **截断行为** ：Solidity 在转换时允许截断，Leo 在可能丢失数据时会失败
  * **类型提升** ：Solidity 在运算中自动提升较小的类型，Leo 需要显式转换


**自动类型提升示例：**


    // Solidity：自动类型提升
    uint8 small = 100;
    uint16 medium = 200;
    uint16 result = small + medium;  // uint8 automatically promoted to uint16


    // Leo：需要显式转换
    let small: u8 = 100u8;
    let medium: u16 = 200u16;
    let result: u16 = (small as u16) + medium;  // Must explicitly cast u8 to u16
    // let invalid: u16 = small + medium;  // 这会失败，类型不匹配

### 引用类型与值类型

**Solidity** 区分值类型和引用类型，需要显式指定数据位置：


    contract TypeSystem {
        uint256[] storageArray;  // 永久存储

        function processData(uint256[] memory memoryArray) public {
            uint256[] storage localRef = storageArray;    // 存储引用
            uint256[] memory localCopy = storageArray;    // 复制到内存

            localRef[0] = 100;   // 修改存储
            localCopy[0] = 200;  // 仅修改内存副本
        }

        function externalCall(uint256[] calldata data) external {
            // Calldata 是只读的，不可修改
        }
    }

**Leo** 仅支持值类型，所有数据在传递时都会被复制：


    program value_types.aleo {
        fn process_data(input_array: [u32; 5]) -> [u32; 5] {
            // 所有数据按值复制
            let local_array: [u32; 5] = input_array;  // 创建副本

            // 可以直接修改单独的元素
            local_array[0u8] = input_array[0u8] + 1u32;
            local_array[1u8] = input_array[1u8] + 2u32;

            return local_array;
        }
    }

**关键差异：**

  * **Leo 限制** ：仅有值类型，无引用类型
  * **内存管理** ：Leo 根据类型语义自动管理内存，无需显式指定存储位置。


## 函数

### 函数类型

Leo 彻底重新构想了函数架构。不同于 Solidity 的可见性修饰符，Leo 使用不同的函数类型：

**Solidity 的方式：**


    contract Example {
        uint256 private data;

        // 外部函数 - 从外部调用
        function publicFunction(uint256 input) external returns (uint256) {
            return internalHelper(input);
        }

        // 内部辅助函数 - 可在继承合约中访问
        function internalHelper(uint256 input) internal view returns (uint256) {
            return input * 2;
        }

        // 私有函数 - 仅可在本合约内访问
        function privateFunction() private pure returns (uint256) {
            return 42;
        }
    }

    contract Child is Example {
        function useParent(uint256 input) public returns (uint256) {
            return internalHelper(input);  // Can access internal functions
            // return privateFunction();   // Cannot access private functions
        }
    }

**Leo 的方式：**


    // 内部 fn：辅助函数（类似于纯函数）—— 在 program 块之外声明
    // 只能计算，不能访问状态，不能进行外部调用
    fn internal_helper(input: u32) -> u32 {
        let doubled: u32 = inline_multiplier(input, 2u32);
        return doubled;
    }

    // 内部 fn：在调用处内联 —— 在 program 块之外声明
    fn inline_multiplier(a: u32, b: u32) -> u32 {
        return a * b;
    }

    program example.aleo {
        // Entry fn: externally callable, off-chain execution
        // All inputs and outputs are private by default, requiring explicit 'public' keyword for on-chain visibility
        fn public_function(input: u32, public pub_input2: u32) -> (public u32, u32) {
            let result: u32 = internal_helper(input);
            let pub_result: u32 = result + pub_input2;
            return (pub_result, result);
        }
    }

**关键差异：**

  * **外部可调用函数** ：在 Leo 中，只有 entry `fn` 函数（通过交易直接调用的函数）可以从外部调用。内部 `fn` 辅助函数只能从程序内部调用
  * **Leo 的纯计算函数** ：Leo 中的内部辅助 `fn` 函数类似于 Solidity 的 `pure` 函数，它们只能执行计算，不能访问状态变量或进行外部调用
  * **Leo 中的可见性** ：与 Solidity 的可见性修饰符（public、private、internal）不同，Leo 的可见性指的是数据隐私，即数据是在链上公开可见还是链下保持私密
  * **函数重载** ：Solidity 支持函数重载（多个同名但参数类型不同的函数），Leo 不允许函数重载。每个函数名在程序中必须唯一，开发者需要为不同参数类型的函数使用不同的名称。


### 返回值

在 Leo 中，返回类型使用箭头（`->`）语法指定，类似于 Rust，这与 Solidity 的 `returns` 关键字不同。

**关键差异：**

  * **返回类型语法** ：Leo 只需要返回类型而非标识符（如 `-> u32`，而 Solidity 用 `returns (uint256 value)`）
  * **返回值处理** ：在 Solidity 中，可以使用逗号选择性地忽略多个返回值（如 `(index, , ) = f();`），而 Leo 要求所有返回值都必须以确切的类型捕获
  * **参数处理** ：类似地，Solidity 允许在函数声明中省略未使用的参数，而 Leo 要求所有参数都必须显式声明其类型


### 修饰符

**Solidity 的修饰符：** Solidity 中的修饰符通常用于修改函数的语义。与 Leo 中的 inline 类似，修饰符体在每个调用处被内联展开。


    contract ModifierExample {
        address public owner;

        modifier onlyOwner() {
            require(msg.sender == owner, "Not the owner");
            _;  // 继续函数执行
        }

        modifier validAmount(uint256 amount) {
            require(amount > 0, "Amount must be positive");
            _;
        }

        function withdraw(uint256 amount) public onlyOwner validAmount(amount) {
            // Function body executes after modifiers
            payable(msg.sender).transfer(amount);
        }
    }

**Leo 的方式：** Leo 没有修饰符，但可以使用辅助函数实现类似功能：


    // 辅助函数 —— 在 program 块之外声明
    fn only_owner(caller: address, owner: address) {
        assert_eq(caller, owner);
    }

    fn valid_amount(amount: u64) {
        assert(amount > 0u64);
    }

    program modifier_example.aleo {
        fn withdraw(amount: u64, owner: address) -> u64 {
            // 辅助函数被显式调用（类似修饰符）
            only_owner(self.caller, owner);
            valid_amount(amount);

            // Function logic continues
            return amount;
        }
    }

### 异步函数

Leo 使用 `final {}` 块进行链上执行：


    program defi_example.aleo {
        mapping user_balances: address => u64;

        fn deposit_and_update(public amount: u64) -> Final {
            // 链下计算
            let new_record: Token = Token {
                owner: self.caller,
                amount: amount,
            };

            // 返回内联 finalization 块用于链上执行
            return final {
                let current_balance: u64 = user_balances.get_or_use(self.caller, 0u64);
                user_balances.set(self.caller, current_balance + amount);
            };
        }
    }

**关键概念：**

  * **Finalization 模式** ：Leo 使用带有内联 `final {}` 块的 `fn` 进行链上执行，`final {}` 块捕获外围 `fn` 作用域中的变量并在链上执行
  * **Final 块** ：`final {}` 块在交易最终确认时在链上执行，允许访问链下计算无法访问的 mapping 值


### 调用限制

Leo 对函数调用层次结构执行严格的规则：

**调用流程：** entry `fn` → internal `fn` | entry `fn` → 外部程序 `fn`

**调用流程规则：**

  * entry `fn` 可以调用内部辅助 `fn` 或外部程序函数。
  * 内部辅助 `fn` 只能调用其他内部辅助函数。
  * 不允许直接/间接递归调用。


**递归：** Solidity 允许函数递归（函数直接或间接调用自身），而 Leo 禁止所有形式的递归调用。


    // Internal helper fn — declared OUTSIDE program block
    fn helper_function(input: u32) -> u32 {
        let processed: u32 = inline_helper(input);      // 有效
        // let invalid: u32 = another_function(input);  // 无效
        return processed;
    }

    fn inline_helper(input: u32) -> u32 {
        let doubled: u32 = inline_doubler(input);       // 有效
        return doubled;
    }

    fn inline_doubler(input: u32) -> u32 {
        return input * 2u32;
    }

    program call_hierarchy.aleo {
        // Entry fn can call: internal helpers, external programs
        fn main_entry(input: u32) -> u32 {
            let result1: u32 = helper_function(input);      // Valid
            let result2: u32 = inline_helper(input);        // Valid
            let result3: u32 = external_program.aleo::some_transition(input); // Valid
            return result1 + result2 + result3;
        }
    }

### Fallback 和 Receive 函数

**Solidity 的 Fallback 和 Receive：**


    contract FallbackExample {
        // Fallback 函数 - 当没有其他函数匹配时调用
        fallback() external {
            // 处理未匹配的函数调用
        }

        // Receive 函数 - 接收 ETH 时调用
        receive() external payable {
            // 处理接收的 ETH
        }

        // 常规函数
        function regularFunction() public {
            // 函数实现
        }
    }

**Leo 的方式：** Leo 没有 fallback 或 receive 函数：


    program fallback_example.aleo {
        // 无 fallback 函数，所有调用必须匹配有效的函数签名
        fn regular_function() {
            // 函数实现
        }

        // 要接收 Aleo Credits，使用 credits.aleo 程序
        fn receive_credits(public amount: u64) -> Final {
            return credits.aleo::transfer_public(self.caller, self.address, amount);
        }
    }

**关键差异：**

  * **无 Fallback** ：Leo 要求所有函数调用必须匹配有效的函数签名
  * **无 Receive** ：程序无需特殊函数来接收 Aleo Credits
  * **程序地址** ：每个 Leo 程序都有一个唯一地址（与用户控制的地址相同）
  * **私有 Records** ：程序无法花费发送给它的私有 records（records 实际上被"销毁"了）
  * **Credit 转账** ：使用 `credits.aleo` 程序的转账函数来发送/接收 credits

### 地址功能与原生代币转账

**Solidity 的地址成员：** Solidity 中的地址有成员函数可以访问其关联的状态，例如：


    contract AddressExample {
        function demonstrateAddressMembers(address payable target) public view returns (uint256, bytes32) {
            // 访问地址属性
            uint256 balance = target.balance;           // 余额（Wei）
            bytes memory code = target.code;            // 地址处的代码（可以为空）
            bytes32 codeHash = target.codehash;         // 地址的 codehash

            return (balance, codeHash);
        }

        function transferMethods(address payable target, uint256 amount) public {
            // 不同的转账方法
            target.transfer(amount);                    // Reverts on failure, 2300 gas stipend
            bool success = target.send(amount);         // Returns false on failure, 2300 gas stipend

            // Low-level calls with adjustable gas
            (bool callSuccess, bytes memory data) = target.call{value: amount}("");
            (bool delegateSuccess, bytes memory delegateData) = target.delegatecall("");
            (bool staticSuccess, bytes memory staticData) = target.staticcall("");
        }
    }

**Leo 的方式：** Leo 地址没有成员函数。Aleo credits 作为程序（`credits.aleo`）在 Aleo 上定义，因此要进行转账和查询余额，使用与查询 Aleo 上任何程序相同的方法，即通过 mapping 公开查询或通过扫描拥有的 records 私下查询。


    import credits.aleo;

    program address_example.aleo {
        // 通过 credits.aleo 程序 mapping 访问余额（公开余额）
        fn get_public_balance(user: address) -> Final {
            return final {
                // 从 credits.aleo 的 account mapping 查询公开余额
                let balance: u64 = credits.aleo::account.get_or_use(user, 0u64);
            };
        }

        // 使用 credits.aleo 程序函数转账 credits
        fn transfer_public_credits(
            to: address,
            amount: u64
        ) -> Final {
            return credits.aleo::transfer_public(self.caller, to, amount);
        }

        fn transfer_private_credits(
            input: credits.aleo::credits,
            to: address,
            amount: u64
        ) -> (credits.aleo::credits, credits.aleo::credits) {
            return credits.aleo::transfer_private(input, to, amount);
        }
    }

**关键差异：**

  * **无地址成员** ：Leo 地址没有 `.balance`、`.code`、`.codehash` 属性
  * **无内置转账方法** ：地址上没有 `.transfer()`、`.send()`、`.call()` 方法
  * **基于程序的转账** ：原生代币转账通过 `credits.aleo` 程序进行
  * **公开与私有** ：余额可以是公开的（mapping）或私有的（records）
  * **统一接口** ：所有程序交互使用相同的调用语法，无论是代币还是其他功能


### 数学运算与溢出处理

**Solidity 的方式：**


    contract MathOperations {
        function checkedMath(uint256 a, uint256 b) public pure returns (uint256) {
            return a + b;  // 默认溢出时回滚（自 0.8.0 起）
        }

        function uncheckedMath(uint256 a, uint256 b) public pure returns (uint256) {
            unchecked {
                return a + b;  // 溢出时环绕
            }
        }

        function modularMath(uint256 a, uint256 b, uint256 n) public pure returns (uint256, uint256) {
            return (addmod(a, b, n), mulmod(a, b, n));
        }
    }

**Leo 的方式：**


    program math_operations.aleo {
        fn checked_math(a: u64, b: u64) -> u64 {
            return a + b;  // Reverts on overflow by default
        }

        fn wrapped_math(a: u64, b: u64) -> u64 {
            return a.add_wrap(b);  // 使用 wrapping 算术溢出时环绕
        }

        fn modular_math(a: u64, b: u64, n: u64) -> (u64, u64) {
            let add_result: u64 = (a + b) % n;
            let mul_result: u64 = (a * b) % n;
            return (add_result, mul_result);
        }
    }

**关键差异：**

  * **环绕运算** ：Leo 使用 `.add_wrap()`、`.sub_wrap()`、`.mul_wrap()` 替代 `unchecked` 块
  * **无内置模运算** ：Leo 没有 `addmod`/`mulmod` 内置函数
  * **显式操作符** ：Leo 要求显式使用环绕操作符

**Solidity 的错误处理：**


    contract ErrorHandling {
        ExternalContract public externalContract;
        error InsufficientBalance(uint256 requested, uint256 available);

        function transfer(uint256 amount) public {
            require(amount > 0, "Amount must be positive");

            if (balance < amount) {
                revert InsufficientBalance(amount, balance);
            }

            try externalContract.riskyCall() returns (string memory) {
                // Success path
            } catch Error(string memory reason) {
                // Handle with reason
            } catch {
                // Handle without reason
            }
        }
    }

**Leo 的错误处理：**


    program error_handling.aleo {
        fn transfer(amount: u64, balance: u64) -> u64 {
            // 简单断言（由于不支持字符串，无自定义消息）
            assert(amount > 0u64);
            assert_neq(amount, 0u64);      // 替代语法
            assert(balance >= amount);

            // 无 try/catch，所有外部调用必须成功
            let result: u64 = external_program.aleo/safe_operation(amount);

        }
    }

## 循环与条件

**Solidity 的控制流：**


    function controlFlow(uint256[] memory items) public pure returns (uint256) {
        uint256 sum = 0;

        // While 循环
        uint256 i = 0;
        while (i < items.length) {
            if (items[i] > 10) {
                continue;  // 跳过此项
            }

            sum += items[i];

            if (sum > 100) {
                break;     // 提前退出
            }

            i++;
        }

        // For loops
        for (uint256 j = 0; j < items.length; j++) {
            // Process items
        }

        return sum;
    }

**Leo 的控制流：**


    fn control_flow(items: [u32; 5]) -> u32 {
        let sum: u32 = 0u32;

        // 仅 for 循环（无 while、do-while）
        for i: u32 in 0u32..5u32 {
            sum += items[i as u8];
            // 无 continue 和 break 语句
        }

        // 三元运算符工作方式相同
        let result: u32 = sum > 100u32 ? 100u32 : sum;
        return result;
    }

**重要限制** ：Leo 目前会执行条件语句的所有分支，然后选择正确的结果。这与仅运行一个分支的典型条件执行不同。此行为可能导致意外问题，尤其是对于可能中断的操作（如除以零）。[ARC-0004](<https://github.com/ProvableHQ/ARCs/discussions/89>) 提议使用标记操作来解决此限制，从而实现正确的 if-then-else 语义。

## 跨程序调用

**Solidity 的动态方式：**


    interface IERC20 {
        function transfer(address to, uint256 amount) external returns (bool);
    }

    contract DynamicCalls {
        function transferTokens(address tokenAddress, address to, uint256 amount) public {
            // 使用接口进行动态合约交互
            IERC20 token = IERC20(tokenAddress);
            bool success = token.transfer(to, amount);
            require(success, "Transfer failed");

            // 低级调用
            (bool callSuccess, bytes memory data) = tokenAddress.call(
                abi.encodeWithSignature("transfer(address,uint256)", to, amount)
            );
            require(callSuccess, "Call failed");
        }
    }

**Leo 的静态调用：**

静态调用（`call`）需要在编译时导入目标程序：


    import credits.aleo;

    program static_calls.aleo {
        fn transfer_credits(
            input: credits.aleo::credits,
            to: address,
            amount: u64
        ) -> (credits.aleo::credits, Final) {
            let tuple: (credits.aleo::credits, Final) = credits.aleo::transfer_private(
                input,
                to,
                amount
            );

            return (tuple.0, final { tuple.1.run(); });
        }
    }

**Leo 的动态调用（[ARC-0009](<https://github.com/ProvableHQ/ARCs/tree/master/arc-0009>)):**

动态调用（`call.dynamic`）在运行时解析目标程序，无需编译时导入。Leo 还支持接口强制语法以实现类型安全的动态调度：


    program dex.aleo {
        // 使用接口语法在运行时调用任何符合 ARC-20 的代币
        fn swap(
            public token_in: identifier,
            public token_out: identifier,
            public amount_in: u128,
            public amount_out: u128,
        ) -> Final {
            let pull: Final = ARC20@(token_in)/transfer_from_public(
                self.signer, self.address, amount_in
            );
            let push: Final = ARC20@(token_out)/transfer_public(
                self.signer, amount_out
            );
            return final { pull.run(); push.run(); };
        }
    }

目标程序和函数名称是**公开的** 电路输入，避免使用敏感的私有数据来选择动态调用目标。详细信息请参阅 [动态调度指南](../../aleo/language/index.mdx#dynamic-dispatch)。

**Solidity 的继承系统：**


    abstract contract Animal {

    contract Dog is Animal {

    contract Cat is Animal {

**Leo 的组合方式：**

Leo 不支持像 Solidity 那样的继承。相反，Leo 使用组合和程序导入来实现类似功能。


    program animal_behaviors.aleo {
        struct AnimalData {
            name: field,
            species: u8, // 1 for dog, 2 for cat
        }

        fn speak(animal: AnimalData) -> field {

            return animal.species == 1u8 ? 100field : 200field; // Dog: "Woof!", Cat: "Meow!"

            // Dogs wag tail, cats purr

    import animal_behaviors.aleo;

    program my_pets.aleo {

        struct AnimalData {
            name: field,
            species: u8,

        record Pet {
            data: AnimalData,

        fn create_dog(name: field) -> Pet {

        fn create_cat(name: field) -> Pet {

### 抽象合约与接口

**Solidity 的抽象类型：**


    // 抽象合约 - 部分函数未实现
    abstract contract Animal {
        string public name;

        constructor(string memory _name) {
            name = _name;
        }

        function speak() public virtual returns (string memory);  // 必须实现

        function eat() public pure returns (string memory) {
            return "Eating...";  // 默认实现
        }

    // Interface - no functions implemented

**Leo 的方式：** Leo 不支持 Solidity 意义上的抽象合约，但现在通过 [ARC-0009](<https://github.com/ProvableHQ/ARCs/tree/master/arc-0009>) 支持用于动态调度的**接口** ：

  * **无抽象合约** ：无法定义部分实现的合约
  * **通过动态调度实现接口** ：Leo 程序可以声明和实现命名接口（如 `ARC20`），调用者可以在运行时使用 `接口@(程序ID)/函数(参数)` 语法调用任何符合接口的程序，无需编译时导入
  * **静态调用的替代方案** ：使用组合和程序导入来实现模块化




    // 任何实现 ARC20 的程序都可以被调用而无需导入
    interface ARC20 {
        fn transfer_public(public recipient: address, public amount: u128) -> Final;
        // ...
    }

    // 在 DeFi 程序中：
    ARC20@(token_id)/transfer_public(recipient, amount);

### 库

**Solidity 的库：**


    library SafeMath {
        function add(uint256 a, uint256 b) internal pure returns (uint256) {
            uint256 c = a + b;
            require(c >= a, "SafeMath: addition overflow");
            return c;
        }
    }

    contract UsingLibrary {
        using SafeMath for uint256;

        function calculate(uint256 a, uint256 b) public pure returns (uint256) {
            return a.add(b);  // 附加到类型的库函数
        }
    }

**Leo 的方式：** Leo 支持原生库 —— 可复用的类型、常量和 `fn` 函数包，它们在**编译时完全内联** 。库没有 transitions、mapping，也没有链上痕迹。使用 `leo new --lib` 创建；入口文件是 `lib.leo`：


    // lib.leo  (包: math)
    const OVERFLOW_GUARD: u64 = 0u64;

    fn safe_add(a: u64, b: u64) -> u64 {
        let result: u64 = a + b;
        assert(result >= a);  // 检查溢出
        return result;
    }

在 `program.json` 中声明：


    {
      "program": "using_library.aleo",
      "dependencies": [
        { "name": "math", "location": "local", "path": "../math" }
      ]
    }

然后以库名作为命名空间访问所有库项 —— 无需 `import` 语句：


    program using_library.aleo {
        fn calculate(a: u64, b: u64) -> u64 {
            return math::safe_add(a, b);
        }
    }

编译器内联每个调用并折叠所有常量。生成的字节码中不包含 `math` 的痕迹。

**关键差异：**

  * **无 Delegatecall** ：Leo 不支持在当前上下文中执行外部代码
  * **无类型附加** ：不能像 Solidity 的 `using` 指令那样将函数附加到类型
  * **编译时内联** ：Leo 库完全在编译时解析 —— 无跨程序调用开销，无 ABI 边界，无链上部署
  * **库 vs 程序** ：库（`lib.leo`）承载共享工具和数据结构；程序（`.aleo`）承载 transitions、mapping 和链上状态

| **库**| **程序**
---|---|---
编译为字节码| 否 — 完全内联| 是
Struct（包括 const generic）| ✓| ✓
Constants| ✓| ✓
函数| 仅 `fn`| 所有变体
Mappings / 存储| —| ✓
链上部署| 否| 是

**Solidity 的 ABI 函数：**


    contract ABIExample {
        function encodeData(uint256 value, string memory text) public pure returns (bytes memory) {
            return abi.encode(value, text);
        }

        function encodePacked(uint256 a, uint256 b) public pure returns (bytes memory) {
            return abi.encodePacked(a, b);
        }

        function decodeData(bytes memory data) public pure returns (uint256, string memory) {
            return abi.decode(data, (uint256, string));
        }
    }

**Leo 的限制：** Leo 没有 ABI 编码/解码功能：

  * **无 bytes 类型** ：Leo 不支持 ABI 操作所需的 `bytes` 类型
  * **无动态编码** ：无法动态编码/解码复杂数据结构
  * **替代方案** ：如需，使用固定大小的数据结构和手动序列化
  * **静态调用** ：所有跨程序调用使用静态定义的接口


## 合约创建与动态部署

**Solidity 的合约创建：**


    contract Factory {
        function createContract(uint256 initialValue) public returns (address) {
            // 使用构造函数参数创建新合约
            MyContract newContract = new MyContract(initialValue);
            return address(newContract);
        }

        function createWithSalt(uint256 initialValue, bytes32 salt) public returns (address) {
            // 使用 CREATE2 创建确定性地址
            MyContract newContract = new MyContract{salt: salt}(initialValue);
            return address(newContract);
        }
    }

**Leo 的限制：** Leo 程序无法创建新程序：

  * **无动态创建** ：程序无法部署其他程序
  * **静态部署** ：所有程序必须通过外部工具部署
  * **无 CREATE2** ：无确定性地址生成，但 Aleo 上的程序已经具有开发者定义的可读名称。
  * **固定架构** ：程序关系必须在部署时建立



::: info[信息]
CREATE2 是 Solidity 中的一个特殊操作码，用于确定性计算创建合约的地址，请参阅 [Solidity 文档](<https://docs.soliditylang.org/en/v0.8.30/control-structures.html#salted-contract-creations-create2>) 了解更多信息。
:::

## 作用域规则

**Solidity 和 Leo 的相似之处：** 两种语言遵循相似的作用域规则：


    // Solidity
    contract ScopeExample {
        uint256 globalVar = 100;

        function scopeDemo() public view returns (uint256) {
            uint256 localVar = 200;
            {
                uint256 blockVar = 300;
                return globalVar + localVar + blockVar;  // All accessible
            }
            // blockVar 在此处不可访问
            return globalVar + localVar;
        }
    }


    // Leo
    program scope_example.aleo {
        fn scope_demo() -> u32 {
            let local_var: u32 = 200u32;

            {
                let block_var: u32 = 300u32;
                local_var = local_var + block_var;  // Both accessible
            }

            // block_var 在此处不可访问
            return local_var;
        }
    }

**关键相似之处：**

  * 变量从声明开始到其包含的 `{}` 块结束前可见


## 内联汇编

**Solidity 的内联汇编：**


    contract AssemblyExample {
        function efficientHash(uint256 a, uint256 b) public pure returns (uint256 result) {
            assembly {
                let hash := keccak256(add(a, b), 0x40)
                result := hash
            }
        }
    }

**Leo 的限制：** Leo 不支持内联汇编：

  * **无类 Yul 集成** ：无法插入底层汇编代码
  * **独立的 IR** ：Aleo Instructions 作为独立语言存在，不可嵌入 Leo 中。
  * **独立程序** ：Aleo Instructions 可以作为独立程序编写在 `.aleo` 文件中，从而实现对程序执行和电路设计的精细低层级控制。有关 Aleo Instructions 的更多详情，请参阅 [Aleo Instructions 概述](../../aleo/aleo/index.mdx)。
