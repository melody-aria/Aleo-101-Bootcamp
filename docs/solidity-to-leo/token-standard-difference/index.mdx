# 代币标准差异

## 介绍

Aleo 有两个活跃的代币标准，各自为网络的不同阶段而设计：

  * **[ARC-21 Token Registry](../../standards/token_registry/index.mdx)** — 现有的单例程序，集中管理所有代币余额。它早于动态调度出现，设计初衷是让 DeFi 程序在编译时无需知晓具体代币即可支持任意代币。
  * **[ARC-20 Token Standard](<https://github.com/ProvableHQ/ARCs/discussions/124>)** — 即将推出的标准（草案中），它利用 [dynamic dispatch (ARC-0009)](<https://github.com/ProvableHQ/ARCs/tree/master/arc-0009>)，让每个代币成为自己的程序，在运行时通过 Leo 的 `Interface@(token_id)/function()` 语法调用。
  * **[ARC-22 Compliant Token Standard](<https://github.com/ProvableHQ/ARCs/discussions/125>)** — 扩展 ARC-20，添加冻结名单执行和合规记录功能，适用于受监管的代币（稳定币、证券代币）。



::: warning[不推荐用于新项目]
代币注册表程序（ARC-21）仍在主网上运行并继续为现有集成服务。但它 **不推荐用于新项目** 。

较新的代币标准（ARC-20 和 ARC-22）正在社区积极讨论中，预计将取代它。新项目应关注这些讨论，并在最终确定后考虑基于即将推出的标准进行构建。
:::

## 快速对比

功能特性| ERC-20| ARC-21 Token Registry| ARC-20
---|---|---|---
代币创建| 部署新合约| 在注册表中注册| 部署实现 `ARC20` 接口的新程序
状态管理| 每个合约独立| 集中式注册表| 每个程序独立
转账机制| 直接合约调用| 注册表程序调用| 通过 `ARC20@(token_id)/function()` 动态调用
隐私| 无| 内置私有转账| 内置私有转账
授权流程| 必需| 可选| 必需（公开）
代币元数据| 在合约中| 在注册表中| 在程序中
供应量管理| 每个合约独立| 集中式注册表| 每个程序独立（可选的 `MintableToken` 扩展）
动态调度| 不适用| 不需要（中心枢纽）| 核心设计要求

## 架构差异

### 程序结构

**ERC-20**

  * 每个代币作为独立的智能合约部署
  * 代币转账需要授权流程
  * 每个代币需要部署新的智能合约


**ARC-21 Token Registry**

  * 单个程序（代币注册表）管理所有代币
  * 基本转账不需要单独的授权流程
  * 新代币通过注册创建，而非部署


**ARC-20**

  * 每个代币是一个独立的已部署程序，声明符合 `ARC20` 接口（Leo 中的 `:InterfaceName`）
  * DeFi 程序在运行时通过动态调度与任何符合标准的代币交互，无需编译时导入
  * 可选的 `MintableToken` 接口扩展 `ARC20`，用于支持铸造/销毁的代币


### 状态管理

**ERC-20**

  * 状态存储在每个代币合约内
  * 合约内直接访问状态
  * 所有状态仅支持公开可见性


**ARC-21 Token Registry**

  * 状态通过注册表程序管理
  * 通过 API 端点或在异步函数中调用 `mappings::get()` 访问状态
  * 同时支持 mapping 的公开状态可见性和 record 的私有状态


**ARC-20**

  * 状态在每个代币自己的程序内管理
  * 公开余额存储在每个程序的 mapping 中；私有余额以 `Token` record 形式持有
  * 符合接口的程序被单独查询，而非通过共享注册表


## 功能差异

完整的 ARC-21 函数签名和参数详情，请参见 [代币注册表程序参考](../../standards/token_registry/index.mdx)。

### 代币创建

**ERC-20** 为每个代币部署一个新合约：


    contract MyToken is ERC20 {
        constructor(string memory name, string memory symbol) ERC20(name, symbol) {
            _mint(msg.sender, 1000000 * 10 ** decimals());
        }
    }

**ARC-21 代币注册表** 通过 `register_token` 在共享注册表程序中注册新代币，提供唯一的 `token_id`、名称、符号、小数位数和最大供应量。无需部署合约。

可选的 `external_authorization_required` 标志启用指定的 `external_authorization_party` 通过限时授权来管控代币转账。适用于归属计划、支出限制或合规控制。使用 `prehook_public` / `prehook_private` 在条件满足后解锁转账。

**ARC-20** 部署一个新程序，声明接口符合性：


    program my_token.aleo: MintableToken {
        record Token {
            owner: address,
            amount: u128,
        }

        fn transfer_public(public recipient: address, public amount: u128) -> Final { ... }
        fn transfer_private(input: Token, to: address, amount: u128) -> (Token, Token) { ... }
        // ... all ARC20 + MintableToken functions
    }

任何 DeFi 程序都可以在运行时按名称调用此代币而无需导入：


    ARC20@(token_id)/transfer_public(recipient, amount);

### 代币转账

**ERC-20**


    function transfer(address to, uint256 amount) public returns (bool)
    function transferFrom(address from, address to, uint256 amount) public returns (bool)

**ARC-21 代币注册表** 提供了多种转账方式：

函数| 描述
---|---
`transfer_public`| 从函数调用者的公开转账
`transfer_public_as_signer`| 从交易签名者的公开转账（在多程序调用中有用）
`transfer_private`| 使用 `Token` record 的私有转账
`transfer_from_public`| 由被授权支出者的公开转账
`transfer_public_to_private`| 将公开余额转换为私有 `Token` record
`transfer_from_public_to_private`| 同上，由被授权支出者发起
`transfer_private_to_public`| 将私有 `Token` record 转换为公开余额

**ARC-20** 通过 `ARC20` 接口定义了一组类似的函数，在运行时使用动态调度调用：

函数| 描述
---|---
`transfer_public`| 公开转账
`transfer_private`| 使用 `Token` record 的私有转账
`transfer_private_to_public`| 私有 `Token` → 公开余额，向发送者发出 `Metadata` 收据
`transfer_public_to_private`| 公开余额 → 私有 `Token`
`shield`| 将签名者的公开余额转换为私有 `Token`
`unshield`| 将私有 `Token` 转回公开
`transfer_from_public`| 由被授权支出者的公开转账
`transfer_from_public_to_private`| 由被授权支出者的公开转私有

从 DeFi 程序动态调用其中任意一个：


    // 适用于任何符合 ARC20 的代币，在运行时解析
    ARC20@(token_id)/transfer_public(recipient, amount);
    ARC20@(token_id)/transfer_from_public(owner, recipient, amount);

使用 `dyn record`（动态 record）进行私有代币操作：


    fn deposit_private(
        private token_record: dyn record,
        public token_id: identifier,
        public amount: u128,
    ) -> (dyn record, Final) {
        let (change, f): (dyn record, Final) =
            ARC20@(token_id)/transfer_private_to_public(token_record, self.address, amount);
        return (change, final { f.run(); });
    }

### 代币铸造

**ERC-20**


    function mint(address to, uint256 amount) public onlyOwner

**ARC-21 代币注册表** 支持 `mint_public` 和 `mint_private`。访问控制基于角色：只有拥有 `MINTER_ROLE` 或 `SUPPLY_MANAGER_ROLE`（或管理员）的地址才能铸造。角色通过 `set_role` / `remove_role` 管理。

**ARC-20** 通过可选的 `MintableToken` 扩展定义铸造：


    fn mint_public(public recipient: address, public amount: u128) -> Final;
    fn mint_private(public recipient: address, public amount: u128) -> (Token, Final);

访问控制由程序自身定义，不由接口强制执行。

### 代币销毁

**ERC-20**


    function burn(uint256 amount) public

**ARC-21 代币注册表** 支持 `burn_public` 和 `burn_private`，采用与铸造相同的基于角色的访问控制（`BURNER_ROLE` 或 `SUPPLY_MANAGER_ROLE`）。

**ARC-20** 通过 `MintableToken`：


    fn burn_public(public amount: u128) -> Final;
    fn burn_private(input: Token) -> Final;

### 代币授权

**ERC-20**


    function approve(address spender, uint256 amount) public returns (bool)
    function allowance(address owner, address spender) public view returns (uint256)

**ARC-21 代币注册表** 使用 `approve_public` 授权支出者额度，使用 `unapprove_public` 撤销或减少额度。

**ARC-20** 使用相同模式（`approve_public` / `unapprove_public`），但有一个关键区别：`approve_public` **累加** 到现有额度而非替换，从而避免了 ERC-20 的授权竞争条件。

### 余额与供应量查询

**ERC-20**


    function balanceOf(address account) public view returns (uint256)
    function totalSupply() public view returns (uint256)

**ARC-21 代币注册表** 通过 RPC mapping 查询直接暴露余额和供应量，无需交易：


    # 余额（复合键：token_id + account）
    GET /{network}/program/token_registry.aleo/mapping/balances/{token_id}_{account}

    # 总供应量（在 TokenMetadata 中）
    GET /{network}/program/token_registry.aleo/mapping/registered_tokens/{token_id}

**ARC-20** 按程序查询：


    # 特定 ARC-20 代币程序中的余额
    GET /{network}/program/{my_token}.aleo/mapping/account/{address}

    # 供应量存储在程序自己的 mapping 中
    GET /{network}/program/{my_token}.aleo/mapping/supply/true

## 总结

[ARC-21](../../standards/token_registry/index.mdx) 代币注册表通过单个程序管理所有代币，提供隐私和基于角色的访问控制，无需 DeFi 程序在编译时知道具体代币。它仍然是现有集成的推荐标准。

[ARC-20](<https://github.com/ProvableHQ/ARCs/discussions/124>) 采用了更去中心化的方式，得益于 [动态调度（ARC-0009）](<https://github.com/ProvableHQ/ARCs/tree/master/arc-0009>)：每个代币是自己的程序，任何 DeFi 程序都可以在运行时通过 `ARC20@(token_id)/function()` 与任何符合标准的代币交互。[ARC-22](<https://github.com/ProvableHQ/ARCs/discussions/125>) 在此基础上扩展了冻结列表执行和调查员可见的合规记录，适用于受监管的发行方。两者目前都是草案，请关注讨论并在最终确定后基于它们进行构建。
