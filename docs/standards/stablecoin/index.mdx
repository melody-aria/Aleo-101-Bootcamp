# 私有稳定币程序

## 概述

私有稳定币程序是一个在 Aleo 区块链上发行和管理隐私保护稳定币的标准。使用此实现的两个稳定币已在主网上线：

### USDCx (`usdcx_stablecoin.aleo`)

![Aleo and Circle partnership](../../../assets/images/aleo-circle-usdcx-83e5dd438ef01f3f742d40bb49984a9f.png)

[`usdcx_stablecoin.aleo`](<https://explorer.provable.com/program/usdcx_stablecoin.aleo>) 是 Aleo 上的首个私有稳定币，由 Aleo 与 [Circle](<https://www.circle.com/blog/usdcx-on-aleo-testnet-via-circle-xreserve>) 合作开发。USDCx 由锁定在以太坊上 Circle xReserve 智能合约中的 **原生 USDC 1:1 支撑** 。当用户向 xReserve 存入 USDC 时，等量的 USDCx 将在 Aleo 上铸造。更多详情请访问 [aleo.org/usdcx](<https://aleo.org/usdcx/>)。

**Circle xReserve** 是一个非托管的智能合约和互操作性基础设施，为 Aleo 上的 USDCx 提供存款和铸造证明。它与两个互补的 Circle 服务配合运作：

  * **Circle Gateway** — 实现统一的 USDC 余额功能
  * **Circle CCTP** （跨链传输协议）— 促进跨区块链资产转移


这些共同使得 USDCx 无需依赖第三方跨链桥就能在支持的区块链之间与 USDC 实现互操作。隐私特性仅适用于 USDCx 停留在 Aleo 上的期间，桥接回其他区块链时将失去这些保护。

### USAD (`usad_stablecoin.aleo`)

![Aleo and Paxos Labs partnership](../../../assets/images/aleo-paxos-labs-usad-5f15b7417e505fd5c5d30ddf550b2acc.png)

[`usad_stablecoin.aleo`](<https://explorer.provable.com/program/usad_stablecoin.aleo>) 是由 [Aleo 网络基金会（ANF）和 Paxos Labs](<https://aleo.org/post/paxos-labs-and-ANF-launch-USAD-Private-Stablecoin/>) 共同开发的隐私保护稳定币。USAD 由 **USDG（Global Dollar）** 支撑，后者是通过 Paxos Labs 的发行框架发行的受监管、机构级别的数字美元。它是首个在结合智能合约能力和增强隐私的第 1 层区块链上发行的美元稳定币。USAD 旨在保持包括参与者身份和交易金额在内的敏感信息机密，同时对监管审查保持透明。更多详情请访问 [aleo.org/usad](<https://aleo.org/usad/>)。

* * *

两个程序共享相同的链上实现，旨在满足监管合规要求，同时利用 Aleo 独特的零知识隐私特性。该程序集成了三个支持组件：

  * 一个 **多签核心** 程序，用于需要多签批准的管理操作
  * 一个 **冻结列表** 程序，用于对代币持有者执行合规控制
  * 一个 **Merkle 树** 程序，用于高效、隐私保护的成员资格证明


此稳定币架构的关键设计目标是允许完全私密的转账，同时仍能实现合规的地址冻结。私有发送者通过提交非成员资格的 Merkle 证明来证明自己不在冻结列表中，而无需在链上暴露地址。

## 合规与隐私设计

### 冻结列表

冻结列表由伴侣程序（`usdcx_freezelist.aleo` / `usad_freezelist.aleo`）管理。冻结列表上的地址将被阻止进行所有公开转账。私有转账在不暴露发送者地址的情况下执行相同的限制：发送者必须提供 Merkle 非成员资格证明，证明其地址不是冻结列表树中的叶子节点。

### ComplianceRecord

每个跨越公开/私有边界的操作（私有铸造、私有销毁和公开到私有的转换）都会向指定的合规官地址发出一个 `ComplianceRecord`。这使得发行者能够为监管目的维护审计跟踪，而无需在链上公开信息。

### 多签管理

管理操作受 `usdcx_multisig_core.aleo`（或 `usad_multisig_core.aleo`）保护，需要多个签名者，由 Aleo 地址和 ECDSA 密钥两者标识，以授权诸如角色更新和程序初始化等敏感变更。

底层模式——由 ECDSA 签名授权的 Aleo 智能合约钱包——在 [`virtual_wallet` Leo 示例](<https://github.com/ProvableHQ/leo-examples/tree/main/virtual_wallet>)中进行了演示。这个示例对于需要借助现有的 ECDSA 密钥基础设施（例如 HSM 或现有的以太坊签名密钥）管理链上资产而无需暴露原生 Aleo 私钥作为授权方的**托管人** 尤其有用。它展示了如何部署一个接受 secp256k1 ECDSA 签名作为其授权机制的程序，使用一个临时的 Aleo 密钥进行交易签名，而 ECDSA 密钥保留实际的支出权限。

## 关键特性

  * 公开和私有的代币余额
  * 基于角色的访问控制（admin、minter、burner）
  * 通过冻结列表执行 Merkle 证明的合规性
  * 基于凭证的私有转账：发送者必须证明自己未被冻结
  * 每次铸造、销毁和公开到私有转账都会发出合规记录
  * 用于紧急停止的暂停机制
  * 多签保护的管理操作


## 如何使用私有稳定币程序

具有 `MINTER_ROLE` 或 `ADMIN_ROLE` 的地址可以通过 `mint_public` 公开铸造代币或通过 `mint_private` 私有铸造代币。具有 `BURNER_ROLE` 或 `ADMIN_ROLE` 的地址可以通过 `burn_public` 或 `burn_private` 销毁代币。

代币持有者可以使用 `transfer_public` 公开转账，或使用 `transfer_private` 私下转账。私有转账要求发送者通过 Provable API 获取当前的冻结列表 Merkle 树，计算两个相邻的 `MerkleProof` 结构体（在字典序上界定其地址，以证明非成员资格），并将这些证明直接传递给 `transfer_private`。

公开余额可以通过 `transfer_public_to_private` 转换为私有代币记录，私有记录可以通过 `transfer_private_to_public` 转换回来。授权机制（`approve_public`、`unapprove_public`、`transfer_from_public`、`transfer_from_public_to_private`）允许第三方代表所有者支出代币。

### 私有转账流程

私有转账在不暴露发送者链上地址的情况下强制执行冻结列表合规。发送者必须通过提供一对字典序上界定其地址的相邻 Merkle 树条目来证明自己不在冻结列表中。

#### 冻结列表 API

要获取私有转账所需的 Merkle 树数据，请查询 Provable 浏览器 API：


    GET https://api.explorer.provable.com/v2/programs/{programID}/compliance/freeze-list

参数| 类型| 描述
---|---|---
`programID`| 路径（字符串）| 稳定币程序 ID，例如 `usdcx_stablecoin.aleo`

**响应：** 一个表示当前冻结列表的 Merkle 树 field 元素数组。使用这些数据计算 `transfer_private` 所需的两个相邻 `MerkleProof` 结构体。

完整 API 参考：[Provable 浏览器 API v2 — 合规冻结列表](<https://docs.explorer.provable.com/docs/api/v2/get-program-program-id-compliance-freeze-list>)

## 数据结构

### Token 记录


    record Token {
        owner: address,
        amount: u128,
    }

### ComplianceRecord 记录


    record ComplianceRecord {
        owner: address,
        amount: u128,
        sender: address,
        recipient: address,
    }

在每次 `mint_private`、`burn_private`、`transfer_public_to_private` 和 `transfer_from_public_to_private` 调用时发出，为合规官提供审计跟踪，而无需在链上公开数据。

### Credentials 记录


    record Credentials {
        owner: address,
        freeze_list_root: field,
    }

必须在调用 `transfer_private` 之前通过 `get_credentials` 获取。证明持有者的地址不在经过验证的 Merkle 根对应的冻结列表中。

### TokenInfo 结构体


    struct TokenInfo {
        name: u128,        // ASCII text encoded as u128 bitstring
        symbol: u128,      // ASCII text encoded as u128 bitstring
        decimals: u8,
        supply: u128,
        max_supply: u128,
    }

### TokenAllowance 结构体


    struct TokenAllowance {
        account: address,
        spender: address,
    }

### MerkleProof 结构体


    struct MerkleProof {
        siblings: [field; 16],
        leaf_index: u32,
    }

在 `get_credentials` 和 `transfer_private` 中用于证明冻结列表中的非成员资格（或成员资格）。树的深度为 16；`siblings` 包含每层一个兄弟节点哈希。

### ChecksumEdition 结构体


    struct ChecksumEdition {
        checksum: [u8; 32],
        edition: u16,
    }

### AdminOp 结构体


    struct AdminOp {
        op: u8,
        threshold: u8,
        aleo_signer: address,
        ecdsa_signer: [u8; 20],
    }

## 映射

`mapping token_info: boolean => TokenInfo;`
以键 `true` 存储代币元数据。有且只有一条记录。

`mapping balances: address => u128;`
将每个地址映射到其公开的代币余额。

`mapping allowances: field => u128;`
将 `TokenAllowance` 结构体的 BHP256 哈希映射到已批准的支出金额。

`mapping address_to_role: address => u16;`
将每个地址映射到其角色位掩码。

`mapping pause: boolean => boolean;`
以键 `true` 存储暂停状态。当为 `true` 时，所有铸造、销毁和转账均被阻止。

## 角色常量

角色以位掩码的形式存储在 `address_to_role` 中。一个地址可以同时拥有多个角色。

角色| 位掩码| 描述
---|---|---
`MINTER_ROLE`| `1u16`| 可以调用 `mint_public` 和 `mint_private`
`BURNER_ROLE`| `2u16`| 可以调用 `burn_public` 和 `burn_private`
`ADMIN_ROLE`| `8u16`| 可以调用包括 `update_role` 在内的所有特权函数

具有 `ADMIN_ROLE` 的地址可以为其他地址分配或撤销角色。管理员不能移除自己的 `ADMIN_ROLE`。

## 函数

### `initialize()`

使用代币元数据初始化稳定币程序并设置初始管理员地址。只能调用一次，且只能由部署者地址调用。

参数| 类型| 描述
---|---|---
`name`| `public u128`| 以 ASCII 编码的代币名称
`symbol`| `public u128`| 以 ASCII 编码的代币符号
`decimals`| `public u8`| 小数位数
`max_supply`| `public u128`| 最大允许供应量
`admin`| `public address`| 初始管理员地址

返回值：`Future`

* * *

### `update_role()`

向目标地址分配新的角色位掩码。调用者必须具有 `ADMIN_ROLE`。管理员不能移除自己的 `ADMIN_ROLE`。

参数| 类型| 描述
---|---|---
`account`| `public address`| 要更新的地址
`role`| `private u16`| 要分配的新角色位掩码

返回值：`Future`

* * *

### `get_credentials()`

通过验证两个相邻的 Merkle 证明（与当前冻结列表根对比）来证明交易签名者不在冻结列表中。该函数确认签名者的地址在字典序上介于两个连续的叶子条目之间，从而建立非成员资格。返回调用 `transfer_private` 所需的 `Credentials` 记录。

参数| 类型| 描述
---|---|---
`proofs`| `private [MerkleProof; 2]`| 在冻结列表树中界定签名者地址的两个相邻证明

返回值：`Credentials`（在当前根上证明非成员资格）、`Future`

* * *

### `get_signing_op_id_for_deploy()`

计算程序部署的签名操作 ID 的实用函数，用于多签管理流程。

参数| 类型| 描述
---|---|---
`checksum`| `private [u8; 32]`| 部署校验和
`edition`| `private u16`| 部署版号

返回值：`field` — `ChecksumEdition` 结构体的 BHP256 哈希，用作签名操作标识符。

* * *

### `mint_public()` / `mint_private()`

向接收者铸造代币。需要 `MINTER_ROLE` 或 `ADMIN_ROLE`。程序不得处于暂停状态，且新供应量不得超过 `max_supply`。`mint_private` 向合规官发出 `ComplianceRecord`。

参数| 类型| 描述
---|---|---
`recipient`| `public address` / `private address`| 接收者地址（`mint_private` 为私有地址）
`amount`| `public u128`| 要铸造的代币数量

返回值：`mint_public` → `Future`；`mint_private` → `ComplianceRecord`、`Token`、`Future`

* * *

### `burn_public()` / `burn_private()`

销毁代币。需要 `BURNER_ROLE` 或 `ADMIN_ROLE`。程序不得处于暂停状态。`burn_private` 向合规官发出 `ComplianceRecord`。

参数| 类型| 描述
---|---|---
`owner`| `public address`| 被销毁代币的地址（仅 `burn_public`）
`input_record`| `Token`| 要从中销毁的 Token 记录（仅 `burn_private`）
`amount`| `public u128`| 要销毁的代币数量

返回值：`burn_public` → `Future`；`burn_private` → 剩余的 `Token`、`ComplianceRecord`、`Future`

* * *

### `transfer_public()` / `transfer_public_as_signer()`

在公开地址之间转账代币。发送者和接收者都不得在冻结列表中。程序不得处于暂停状态。`transfer_public_as_signer` 使用 `self.signer` 作为发送者而非 `self.caller`，从而在程序调用链中使用，同时将借方归于原始交易签名者。

参数| 类型| 描述
---|---|---
`recipient`| `public address`| 接收者地址
`amount`| `public u128`| 转账金额

返回值：`Future`

* * *

### `approve_public()` / `unapprove_public()`

`approve_public` 授予支出方代表调用者转账代币的能力，按指定金额增加授权额度。`unapprove_public` 减少或撤销该授权额度。

参数| 类型| 描述
---|---|---
`spender`| `public address`| 要授权或取消授权的地址
`amount`| `public u128`| 要从授权额度中增加或减去的金额

返回值：`Future`

* * *

### `transfer_from_public()`

使用预先批准的授权额度从所有者向接收者转账代币。所有者和接收者都不得在冻结列表中。程序不得处于暂停状态。

参数| 类型| 描述
---|---|---
`owner`| `public address`| 要借方的地址
`recipient`| `public address`| 要贷方的地址
`amount`| `public u128`| 转账金额

返回值：`Future`

* * *

### `transfer_public_to_private()` / `transfer_from_public_to_private()`

将公开余额转换为私有 `Token` 记录，向合规官发出 `ComplianceRecord`。`transfer_from_public_to_private` 使用预先批准的授权额度代表所有者进行操作。两者都要求发送者/所有者不在冻结列表中，且程序未暂停。

参数| 类型| 描述
---|---|---
`owner`| `public address`| 要公开借方的地址（仅 `transfer_from_public_to_private`）
`recipient`| `private address`| 私有接收者地址（在链上不可见）
`amount`| `public u128`| 要转换的金额

返回值：`ComplianceRecord`、`Token`、`Future`

* * *

### `transfer_private()`

在两个私有 `Token` 记录之间转账代币。发送者必须提供两个相邻的 `MerkleProof` 结构体，证明其地址不在当前冻结列表中。程序不得处于暂停状态。

参数| 类型| 描述
---|---|---
`recipient`| `private address`| 接收者地址（在链上不可见）
`amount`| `private u128`| 转账金额（在链上不可见）
`input_record`| `Token`| 发送者的 Token 记录
`proofs`| `private [MerkleProof; 2]`| 证明发送者不在冻结列表中的 Merkle 证明

返回值：发送者剩余的 `Token`、接收者的新 `Token`、`Future`

* * *

### `transfer_private_to_public()`

将私有 `Token` 记录转换为公开余额。程序不得处于暂停状态。

参数| 类型| 描述
---|---|---
`recipient`| `public address`| 公开接收者地址
`amount`| `public u128`| 要转换的金额
`input_record`| `Token`| 发送者的私有 Token 记录

返回值：发送者剩余的 `Token`、`Future`
