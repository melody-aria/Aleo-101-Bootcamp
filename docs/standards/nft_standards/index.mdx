# NFT 标准

## 概述

NFT 标准文档概述了在 Aleo 区块链上实现非同质化代币（NFT）的规范。该标准源自 [ARC-0721 提案](<https://github.com/ProvableHQ/ARCs/discussions/79>)，并通过社区投票在 [Aleo 治理](<https://vote.aleo.org/p/721>)上正式获批。

随着 [动态分发（ARC-0009）](<https://github.com/ProvableHQ/ARCs/tree/master/arc-0009>)的最终确定，程序可以在运行时调用其他程序而无需编译时导入，从而解决了以前需要集中注册表方法的可组合性挑战。已提议的 [NFT Registry 程序（ARC-722）](<https://github.com/ProvableHQ/ARCs/discussions/80>)将作为 NFT 集合的中心枢纽，并实现 NFT 与 DeFi 应用之间的互操作性，但目前仍处于提案阶段。

## 关键特性

该标准利用 Aleo 独特的隐私特性提供：

  * 私有或公开的代币所有者可见性
  * 与代币关联的私有或公开数据
  * 灵活的链上和链下数据存储选项


## 数据存储选项

### 链上数据 vs 链下数据

该标准允许以三种方式存储 NFT 数据：

  1. **链上数据**

     * 数据直接存储在区块链上
     * 可以是完整数据或数据的哈希值
  2. **链下数据**

     * 减少网络上的存储费用
     * 通常以指向外部元数据的 URI 形式存储
     * 可以与链上数据结合用于混合方案
  3. **混合方案**

     * 链上和链下存储的组合
     * 公开数据存储在链上（例如 NFT 所有权）
     * 私有数据存储在链下（例如 NFT 数据）


## 数据结构

### NFT 记录


    record NFT {
        private owner: address,
        private data: data,
        private edition: scalar,
    }

#### NFT 记录字段

  * `owner`: NFT 所有者的私有地址
  * `data`: 与 NFT 关联的私有数据
  * `edition`: 用于唯一性和隐私性的标量值


### NFT 查看记录


    record NFTView {
        private owner: address,
        private data: data,
        private edition: scalar,
        private is_view: bool
    }

#### NFT 查看记录字段

  * `owner`: NFT 所有者的私有地址
  * `data`: 与 NFT 关联的私有数据
  * `edition`: 用于唯一性和隐私性的标量值
  * `is_view`: 用于区分 NFTView 和 NFT 的布尔标志（始终为 true）


### Data 结构体


    struct attribute {
        trait_type: [field; 4],
        _value: [field; 4],
    }

    struct data {
        metadata: [field; 4], // URI of offchain metadata JSON
        // (optional) name: [field; 4],
        // (optional) image: [field; 16],
        // (optional) attributes: [attribute; 4],
        // (optional) ...
    }

#### Data 结构体字段

  * `metadata`: 指向链下元数据 JSON 的 URI
  * `name`: NFT 的可选名称
  * `image`: 可选的图像数据
  * `attributes`: 可选的属性数组


此类链下元数据 JSON 的示例可在此处[查看](<https://aleo-public.s3.us-west-2.amazonaws.com/testnet3/privacy-pride/1.json>)。

### NFT Content 结构体


    struct nft_content {
        data: data,
        edition: scalar
    }

#### NFT Content 结构体字段

  * `data`: 与 NFT 关联的数据
  * `edition`: NFT 的版号


## 映射

`mapping nft_commits: field => bool;`
将 NFT commits 映射到其存在状态的映射。

`mapping nft_owners: field => address;`
将 NFT commits 映射到其公开所有者的映射。

`mapping nft_contents: field => nft_content;`
将 NFT commits 映射到其公开内容的映射。

## 函数

### `commit_nft()`

#### 描述

通过将 NFT 的数据和版号提交到一个 field 中来为 NFT 创建唯一标识符。

#### 参数

  * `nft_data: data`：NFT 的数据
  * `nft_edition: scalar`: NFT 的版号


#### 返回值

  * `field`：NFT commit 标识符


### `transfer_private_to_public()`

#### 描述

将私有拥有的 NFT 转换为公开所有权。

#### 参数

  * `private nft: NFT`：要转换的 NFT 记录
  * `public to: address`：公开的接收者地址


#### 返回值

  * `NFTView`：NFT 查看记录
  * `Future`：用于完成转账的 Future


### `publish_nft_content()`

#### 描述

发布 NFT 内容使其可公开访问。

#### 参数

  * `public nft_data: data`：要发布的 NFT 数据
  * `public nft_edition: scalar`：要发布的版号


#### 返回值

  * `Future`：用于完成发布的 Future


### `update_edition_private()`

#### 描述

更新私有 NFT 的版号以重新混淆其内容。

#### 参数

  * `private nft: NFT`：要更新的 NFT 记录
  * `private new_edition: scalar`：新版号


#### 返回值

  * `NFT`：更新后的 NFT 记录
  * `Future`：用于完成更新的 Future


## 字符串编码

NFT 严重依赖字符串的使用，无论是用于链下数据的 URL 还是数据本身。该标准指定了以下字符串编码方式：


    // Leo
    string: [field; 4],

    // Aleo instructions
    string as [field; 4u32];

数组的长度可以自由调整以匹配集合所需的最大字符数。选择 field 类型的原因是它在相同约束条件下提供的数据量几乎是 u128 的两倍。

对于 JavaScript/TypeScript 应用程序，在 JavaScript 字符串和 Aleo 明文之间进行转换的示例可在 [ARC-721 实现](https://github.com/zsociety-io/aleo-standard-programs/blob/main/arc721/utils/strings.mdx)中找到。

## 隐私特性

该标准通过多种机制实现隐私：

### 所有权隐私

  * 通过 Aleo 记录实现私有所有权
  * 可通过 `nft_owners` 映射启用公开所有权
  * 程序可以拥有 NFT 而不暴露其数据


### 数据隐私

  * NFT 数据在记录中默认保持私有
  * `edition` 标量确保唯一性而不暴露数据
  * NFT commit 作为唯一标识符而不暴露底层数据


### 重新混淆

NFT 可以通过两步流程进行重新混淆：

  1. 转回私有所有权
  2. 使用 `update_edition_private()` 更新版号




    fn update_edition_private(
        private nft: NFT,
        private new_edition: scalar,
    ) -> (NFT, Final) {
        let out_nft: NFT = NFT {
            owner: nft.owner,
            data: nft.data,
            edition: new_edition,
        };
        let nft_commit: field = commit_nft(nft.data, new_edition);

        return (out_nft, final {
            assert(nft_commits.contains(nft_commit).not());
            nft_commits.set(nft_commit, true);
        });
    }

重要的隐私注意事项：

  * 之前的 NFT commit 保留在映射中以防止暴露数据关系
  * 新版号必须是唯一的
  * 此过程在创建新的公开标识符的同时保持了数据隐私


## 授权

该标准包含一个授权机制，允许指定地址代表所有者转移 NFT。授权系统同时支持集合级和单个 NFT 的授权：


    struct approval {
        approver: address,
        spender: address
    }

    mapping for_all_approvals: field => bool;
    // Approval hash => Is approved

    mapping nft_approvals: field => field;
    // NFT commit => Approval hash

授权系统提供两个主要函数：

  1. `set_for_all_approval`：允许所有者授权支出方使用集合中的所有 NFT
  2. `approve_public`：允许所有者授权支出方使用特定的 NFT


一旦获得授权，支出方可以使用 `transfer_from_public` 将 NFT 从授权者转移到接收者地址。

## 设置

集合级设置通过映射进行管理：


    mapping general_settings: u8 => field;
    // Setting index => Setting value

可用的设置索引及其用途：

  * `0u8`: 可铸造的 NFT 数量（所有版号）
  * `1u8`: 可铸造的总 NFT 数量（首发版号）
  * `2u8`: NFT 的符号
  * `3u8`: NFT 的基础 URI，第 1 部分
  * `4u8`: NFT 的基础 URI，第 2 部分
  * `5u8`: NFT 的基础 URI，第 3 部分
  * `6u8`: NFT 的基础 URI，第 4 部分
  * `7u8`: 管理员地址哈希


这些设置允许对 NFT 集合的属性进行精细控制，包括铸造限制、元数据位置和管理控制。

## 实现说明

  1. 该标准兼容用于同质化代币名称和符号的 ARC21 标准。

  2. 对于数据可以公开的集合（"可发布集合"），该标准提供了发布和管理公开内容的机制，同时在需要时保留重新混淆数据的选项。

  3. [NFT Registry 程序（ARC-722）](<https://github.com/ProvableHQ/ARCs/discussions/80>)被提议作为 NFT 集合的中心枢纽（类似于 [Token Registry 程序](../token_registry/index.mdx)如何为同质化代币工作），允许通过 (registry_program_id, collection_id) 对来识别具有不同数据结构的多个实现。随着 [动态分发（ARC-0009）](<https://github.com/ProvableHQ/ARCs/tree/master/arc-0009>)的推出，DeFi 程序也可以在运行时直接与单个 NFT 程序交互，而无需注册表。
