# SnarkOS Contribution

This checklist provides a step-by-step guide for restarting the Aleo network with new features merged. Follow these steps to ensure a smooth process.

## 分支概述

[**staging 分支**](<https://github.com/AleoNet/snarkOS>)

  * 此分支作为集成的候选发布区域和更改的初始测试环境，然后再提升到主网分支。


[**mainnet 分支**](<https://github.com/AleoNet/snarkOS/tree/mainnet>)

  * 生产分支，只有稳定且经过充分测试的更改才会合并到此。用于创建生产发布版本。它始终是候选发布提交的直接镜像。


## 网络概述

### 开发网

  * 初始提议的更改在开发网上实现和测试。
  * Snarkops —— 旨在提供管理 SnarkOS 和参与 ANF 的 Canary 网的指南和脚本。


### Canary 网（运行 staging 分支）

  * 更改从开发网合并到 Canary 网进行测试和验证。
  * Canary 网用于在潜在将验证者绑定到测试网 Beta/主网之前引入更多验证者。
  * 验证者由 ANF 绑定。
  * snarkOS GitHub 标签标准: `canary-v*`。 [标签链接](<https://github.com/AleoNet/snarkOS/tags>)。
  * [Canary 网浏览器](<https://vision.snarkos.net/?blocks>)


### 测试网 Beta（运行 mainnet 分支）

  * 开放的公共网络，用于在类似生产环境中测试应用而无需产生成本。
  * 验证者由 Aleo 网络基金会绑定。
  * ANF 和 Provable 最初运行验证者。
  * Demox Labs 和 Puzzle 运行公开水龙头
  * snarkOS GitHub 标签标准: `testnet-beta-v*` [标签链接](<https://github.com/AleoNet/snarkOS/tags>)。
  * [测试网 Beta 浏览器](<https://vision.snarkos.net/?blocks>)


### 主网（运行 mainnet 分支）

  * 全面生产部署前的最终测试阶段。
  * 旨在成为新代码和/或验证者引入的 “最后一站”。
  * 将在成为正式主网之前与测试网 Beta 并行运行。
  * snarkOS GitHub 标签标准: `mainnet-v*` [标签链接](<https://github.com/AleoNet/snarkOS/tags>)。
  * [主网浏览器](<https://vision.snarkos.net/?blocks>)


## 贡献工作流程 [ snarkOS ]

![SnarkOS 贡献流程图](../../../assets/images/snarkos_contribute_flow-27e629c71a985fe466be9b5817a74c0f.png)

**1\. 复刻仓库**

  * 复刻仓库：从 staging 分支复刻到你自己的 GitHub 账户。
  * 在本地克隆你的复刻。




    git clone git@github.com:AleoNet/snarkOS.git
    git remote add upstream git@github.com:AleoNet/snarkOS.git

**2\. 切换到基础分支**


    git switch staging

**3\. 创建功能分支**

  * 从你复刻的 main 分支创建功能分支。




    git checkout -b feat/my-branch

**4\. 开发你的功能/修复并测试**

  * 在你复刻的仓库中开发你的功能或修复。
  * 运行 `./snarkOS/devnet.sh`
  * 确保批准重新安装 snarkOS 二进制文件的选项，以便使用你当前的本地 snarkOS 代码进行测试。
  * 观察网络是否正常推进区块，并发送一些交易以进一步确认网络的稳定性。
  * 运行与你的功能/修复相关的特定测试。


**5\. 将你的代码推送到修复分支**

  * 使用有意义的提交信息提交更改，清楚描述更改及其目的。
  * 将更改推送到你的复刻。




    git add .
    git commit -m "Add detailed description of the changes"
    git push

**6\. 向你的复刻提交 PR**

  * 从你的功能分支向你复刻的 main 分支创建 PR。
  * 这将触发你复刻中的 CI 流水线运行自动化测试。


**7\. 内部代码审查**

  * 在你的复刻仓库中向你的团队请求内部代码审查。
  * 团队成员审查代码，提出修改建议，并在符合质量标准时批准 PR。
  * CI 流水线会再次运行，以确保审查过程中没有引入新的问题。


**8\. 向主 AleoNet/snarkOS 仓库提交 PR**

  * 内部批准后，从你复刻的 main 分支向主 AleoNet/snarkOS 仓库的 staging 分支提交 PR。
  * 这将触发主 CI 流水线再次运行所有相关测试和检查，以确保代码的稳定性和兼容性。


**9\. 核心团队审查**

  * 核心团队成员在主 AleoNet/snarkOS 仓库中审查 PR。
  * 如果 CI 流水线通过且审查成功，则批准 PR。
