## 问题

**Q1. Leo 中的 "Private by Default"（默认隐私）语义是什么？**

A:Leo 的 "默认隐私" 是其核心设计哲学，与传统编程语言（如 Java、Solidity）的 "默认可见性" 有本质区别 —— 它不是简单的代码访问权限控制，
而是密码学级别的数据隐私保证。在 Leo 中，所有未显式声明可见性的元素，都会自动被设置为 private

**Q3. Aleo record 中 owner 字段的作用是什么？**

A:owner 是 Aleo Record 最核心、唯一强制内置的字段，它不是一个普通的业务属性，而是整个 Aleo 隐私所有权模型、交易执行和双花防护的基石。所有自定义 Record 都会自动继承这个字段，无法删除、无法修改类型。

**Q4. 程序中的 final 是什么？**

A:final 是 Leo 语言最核心的特性之一，也是 Aleo"混合隐私计算模型" 的关键实现。它解决了一个核心问题：如何在不泄露隐私的前提下，安全地更新链上公共状态。
简单来说：
transition：在用户本地执行，做隐私计算，生成零知识证明
final：在区块链上执行，做公共状态更新，所有节点共识验证

**Q5. 如何创建 helper functions（辅助函数）？**

A:辅助函数是 Leo 中用于封装可复用逻辑的普通函数，它们不直接处理交易或链上状态，而是为 transition 和 final 函数提供通用计算能力，帮助你减少代码重复、提高可读性和可维护性。
一、基本语法与定义
1. 函数定义格式
Leo 辅助函数使用 function 关键字定义，语法非常简洁：

// 基本格式
function 函数名(参数名: 参数类型) -> 返回值类型 {
    // 函数体
    return 返回值;
}
3. 示例

// 计算两个数的和
function add(a: u64, b: u64) -> u64 {
    return a + b;
}

// 计算平方
function square(x: u64) -> u64 {
    return x * x;
}

**Q6. helper functions 能否创建 records？**

A:辅助函数（模块级 fn）是纯计算函数，绝对不能创建或返回 Record 类型。这是 Leo 语言最核心的设计限制之一，与 Record 的本质和 Aleo 的执行模型密切相关。
一、为什么辅助函数不能创建 Record？
1. Record 的特殊本质
Record 不是普通的结构体，它是 Aleo 区块链上私有状态的基本单元，具有以下特殊属性：
每个 Record 都有唯一的承诺（commitment）和序列号（nullifier）
只能被消费一次，消费后会在链上标记为已花费
必须由特定的地址拥有，只有所有者才能消费
这些属性决定了 Record 只能在交易执行的上下文中创建，而不能在普通的计算函数中创建。

**Q7. constructor 的目的是什么？**

A:constructor 是 Aleo 程序的唯一初始化函数，它在程序第一次部署到区块链时自动执行一次，且永远只能执行一次。它是程序生命周期的起点，负责设置程序的初始状态和配置。
核心本质与目的
constructor 的核心目的是：在程序对所有用户开放之前，完成一次性的初始化工作，确保程序从一个已知的、正确的状态开始运行。
它解决了一个关键问题：如何在程序部署时，安全地设置那些只能由部署者设置一次的初始值（比如管理员地址、代币总供应量、初始配置参数等）。

**Q8. 如何组合多个 interfaces（接口）？**

A:Leo 支持接口多继承和多接口实现两种方式来组合多个接口，这是实现代码模块化、复用和多态的核心机制。接口组合允许你将多个细粒度的功能契约合并成一个更大的契约，或者让一个程序同时满足多个不同的接口要求。
核心概念
在 Leo 中，接口（interface）是一组函数签名的集合，定义了程序必须实现的公共契约。组合接口有两种标准方式：
继承式组合：定义一个新接口，继承多个现有接口的所有函数
直接实现式组合：让一个程序直接实现多个独立的接口
两种方式都能达到 "一个程序拥有多个接口功能" 的效果，适用于不同的场景。

**Q9. record interface 中 `..` 的含义是什么？**

A:在 Leo 的 Record Interface / Record / Struct 中，.. 是字段展开（Spread / 剩余继承）语法，
核心作用是：批量继承 / 复用已有结构的全部字段，无需逐个手写，分为接口定义和实例构造两种核心场景。

---

**Q10. 何时使用 dyn record（动态 record）？**

A:dyn record 是 Leo 中的动态记录类型，基于 record interface 实现Record 多态，用于指代任意实现了指定接口的 Record。
必须使用 dyn record 的场景（核心答案）
统一处理多种不同的 Record 类型
当程序中有多个自定义 Record，且它们都实现了同一个 record interface 时，用 dyn record 作为通用类型，兼容所有这些 Record。
编写通用、可复用的函数逻辑
不绑定具体的 Record 类型，让函数支持任意符合接口规范的 Record，提升代码模块化和复用性。
实现 Record 的多态调用
无需关心具体的 Record 实现，仅通过接口契约调用通用方法 / 访问字段。

**Q11. storage vector 支持的核心操作有哪些？**

A:storage vector 是 Leo 中链上持久化的动态向量（动态长度、固定类型的链上存储结构），专门用于存储程序的动态链上状态。
核心操作（必背，考试 / 作业直接写）
len()：获取向量的元素总长度
is_empty()：判断向量是否为空
push(item)：向向量尾部添加元素
pop()：删除向量尾部元素并返回该元素
get(index)：根据索引只读读取元素
set(index, item)：修改指定索引的元素值
insert(index, item)：在指定索引位置插入元素
remove(index)：删除指定索引的元素
clear()：清空向量内所有元素
